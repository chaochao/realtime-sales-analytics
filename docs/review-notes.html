Q2 — How does correction learning work?

When a user types a natural-language query, the LLM parses it into a structured draft filter. The resolver then turns that draft into real DB values — first checking stored corrections, then falling back to substring matching. When the user confirms an interpretation that involved an ambiguous term, that mapping is saved so the same term resolves automatically next time.

Happy path

1. User query "Show john's deals in the West"
↓
2. LLM parse → draft filter NL extraction { salesRep: "john", region: "West" }
↓
3. Resolver checks corrections DB first resolver.ts:34 lookupCorrection("john", "salesRep") → null (first time)
↓
4. Substring match → ambiguity resolver.ts:40 "john" matches: John Smith, John Doe → asks user to pick
↓
5. User selects "John Smith" → confirmation prompt "I'm reading this as: salesRep = John Smith. Is that right?"
↓
6. User clicks "Yes, apply" → correction saved ChatPanel.tsx:110 POST /api/correction { term: "john", field: "salesRep", resolvedValue: "John Smith" }
↓
7. Next query: "show john's Q2 deals" lookupCorrection("john", "salesRep") → "John Smith" immediately, no prompt

Code: saving & looking up corrections

// corrections.ts — upsert so the same term can be updated later
export async function saveCorrection(term, field, resolvedValue) {
  await prisma.correction.upsert({
    where: { term_field: { term: normalizeTerm(term), field } },
    update: { resolvedValue },        // overwrites stale mapping
    create: { id, term, field, resolvedValue, createdAt },
  });
}

// resolver.ts — correction check comes BEFORE substring matching
const corrected = await lookupCorrection(normalizeTerm(term), field);
if (corrected) {
  resolved[field] = corrected;
  continue;        // skip ambiguity entirely
}

The tricky edge cases

Edge case 1 — Correction already exists, user wants a different one Once "john → John Smith" is stored, the resolver short-circuits before ambiguity runs. The user has no button to click. Solution: the "No" button triggers a correction flow — it saves correctionBase and correctionDraft, lets the user retype, and if the newly resolved value differs from the draft, calls /api/correction to overwrite the old mapping. (ChatPanel.tsx:77–88)
Edge case 2 — Ambiguity resolved via button, not free text When the user clicks a candidate button (not the Yes/No confirm), the term is still ambiguous by definition, so the correction payload is attached to pendingConfirm at that point. "Yes, apply" then fires the save. (ChatPanel.tsx:100–103)
Edge case 3 — Cross-field fallback If "john" matches no salesRep, the resolver checks the customer field instead (and vice versa). This silently promotes the field, which is then reported in the interpretation string. (resolver.ts:43–50)

State variables that drive the flow

// ChatPanel.tsx — four pieces of state coordinate the confirmation + correction flow
pendingConfirm    // holds filter + interpretation awaiting Yes/No
correctionBase    // the last confirmed filter; carried into the next query on "No"
correctionDraft   // the raw LLM parse; used to detect if the value changed
pendingAmbiguity  // the unresolved term when the agent asks the user to pick

Q4 — How would you scale this beyond a single instance?

The current design has two hard single-instance constraints: an in-process event bus and a file-locked SQLite database. Both are intentional trade-offs for zero-infrastructure local development. Scaling to multiple instances requires replacing each with a shared equivalent.

Current architecture

// events.ts — globalThis keeps one bus across Next.js hot reloads
const g = globalThis as { __bus?: EventEmitter };
const bus = g.__bus ?? (g.__bus = new EventEmitter());

// publishTransaction → bus.emit("event", payload)
// subscribe        → bus.on("event", handler)  (one listener per open SSE tab)

When POST /api/transactions fires, it calls publishTransaction. Every open SSE connection has registered a handler via subscribe. This works perfectly within one process — but a second instance has its own globalThis, so transactions created on instance A are invisible to tabs connected to instance B.

What breaks at scale and how to fix it

Today (single instance)

  • Event bus: EventEmitter in memory
  • Database: SQLite via Prisma + better-sqlite3
  • SSE clients: all live in the same process
  • Corrections: SQLite rows, read/written locally

Multi-instance path

  • Event bus → Redis pub/sub (publish on one node, all nodes subscribe)
  • Database → Postgres (shared, concurrent-safe)
  • SSE routes stay the same — just swap subscribe() source
  • Corrections table moves with the DB, no logic change

Minimal code change needed

// Today — subscribe is a local EventEmitter listener
export function subscribe(handler) {
  bus.on("event", handler);
  return () => bus.off("event", handler);
}

// After — subscribe listens on a Redis channel instead
export function subscribe(handler) {
  const sub = redis.duplicate();
  sub.subscribe("transactions", (msg) => handler(JSON.parse(msg)));
  return () => sub.unsubscribe();
}

// publishTransaction changes similarly — redis.publish("transactions", JSON.stringify(event))
// Everything else (SSE route, analytics, drift) stays identical.

What does NOT need to change

The SSE route handler, analytics computation, drift detection, and correction lookup are all stateless relative to the bus. They receive an event or a DB query result and return a value. Swapping the bus and DB underneath them requires no logic changes — only the two transport functions (publishTransaction, subscribe) and the Prisma datasource URL.

Other limitations worth mentioning

Static FX rates Currency conversion uses hardcoded rates in currency.ts. In a production multi-tenant system these would come from a live rates API or a daily-updated table.
No auth / single user Corrections are global — there's no user ID on the Correction row. Multi-user would require scoping corrections by userId and adding auth middleware.

Step-by-step: How correction learning works

This is the full walkthrough of what happens from the moment a user types a query to the moment a correction is stored and silently applied on future queries.

Step 1 — User types a query e.g. "Show me john's deals in the West"
↓
Step 2 — LLM parses the text into a draft filter POST /api/query { text: "john's deals in the West" }
LLM returns: { salesRep: "john", region: "West" }
↓
Step 3 — Resolver checks the corrections table first (field-scoped) lookupCorrection("john", "salesRep")
First time → returns null. Skip to step 4.
Subsequent times → returns "John Smith". Jump straight to step 8.
Note: does NOT check the customer field here — corrections are keyed by (term, field). A "john → John Smith" correction saved under customer is a separate row and is not consulted.
↓
Step 4 — Substring matching against known DB values Exact match first; if none, substring.
"john" matches: John Smith, John Doe → 2 candidates → ambiguity
↓
Step 5 — Agent asks the user to pick (clarify UI) Chat shows: "Did you mean one of these for salesRep 'john'?"
Two buttons appear: John Smith   John Doe
↓
Step 6 — User clicks a candidate e.g. clicks John Smith.
pendingConfirm is set with { filter: { salesRep: "John Smith", region: "West" }, correction: { term: "john", field: "salesRep" } }
↓
Step 7 — Agent shows confirmation prompt "I'm reading this as: salesRep = John Smith, region = West. Is that right?"
Two buttons: Yes, apply   No
↓ (two paths)

Path A — User clicks "Yes, apply"

  • Filter is applied to the dashboard
  • POST /api/correction { term: "john", field: "salesRep", resolvedValue: "John Smith" }
  • Saved via correction.upsert (overwrites if exists)
  • Next query with "john" → step 3 returns "John Smith" immediately

Path B — User clicks "No" OR just types a follow-up

  • Clicking No sets correctionBase and correctionDraft explicitly, clears the prompt
  • Typing while the prompt is still visible works too — send() reads pendingConfirm?.filter and pendingConfirm?.draft as fallbacks (line 54–56)
  • Either way, the next query is sent with the resolved filter as context and the original LLM term is preserved for correction detection
  • If the newly resolved value differs from the draft, a new correction is saved automatically
↓
Step 8 — Future queries auto-resolve "john" → lookupCorrection → "John Smith" → no ambiguity prompt, no button, no confirmation.
The user sees the result immediately.

-- Composite unique key: (term, field)
-- So "john" as salesRep and "john" as customer are separate rows

The "No" path in detail

When the user clicks No, the UI saves two pieces of state: correctionBase (the last resolved filter) and correctionDraft (the raw LLM output before resolution). The next message the user sends is passed to /api/query with correctionBase as context, so the LLM refines rather than restarts. After the new resolution, the code compares the new resolved value against correctionDraft field by field — if they differ, that's the user's correction, and it's saved to the DB.

Key invariant

Corrections are checked before matching, not after. This means a stored correction completely bypasses ambiguity detection. Once "john → John Smith" is saved, the substring match that found two candidates never runs. The only way to change it is through the "No" path, which explicitly re-runs the flow and overwrites the stored row via upsert.
Corrections are field-scoped — cross-field lookup does not apply. The correction lookup uses the exact field being resolved: lookupCorrection("john", "salesRep"). It does not fall back to checking the customer correction for the same term. Cross-field logic only kicks in later, during substring matching, when zero salesRep candidates are found. A "john → John Smith" correction saved under customer is a separate DB row and is invisible to the salesRep resolution path.

When does the "Did you mean…?" UI appear?

The disambiguation UI shows when a text term does not resolve to exactly one known value and isn't already covered by a saved correction. Concretely: after the LLM produces a draft filter, resolveFilter matches each text field (salesRep, region, customer, currency) against the known DB values. Anything other than a single clean match becomes an ambiguity, the API returns status: "clarify", and ChatPanel renders the bubble + candidate buttons.

The matching logic resolver.ts:9

// matchCandidates — exact first, then substring fallback
const exact = known.filter((k) => k.toLowerCase() === t);
if (exact.length) return exact;
return known.filter((k) => k.toLowerCase().includes(t));

// "john" has no exact rep, so substring runs →
// matches "John Doe" AND "John Smith" → 2 candidates → ambiguity

The three outcomes

Term matches 2+ known values resolver.ts:52 "Did you mean one of these for salesRep 'john'?" — with candidate buttons. This is the screenshot case ("john" → John Doe, John Smith).
·
Term matches 0 values (no cross-field fallback) ChatPanel.tsx:68 "No salesRep matching 'john' found." — no buttons, since there are no candidates to offer.
·
Term matches exactly 1 value resolver.ts:41 No clarify UI — it resolves silently and goes straight to the Yes/No confirmation prompt.

What suppresses the UI even when a term is ambiguous

A saved correction short-circuits it. lookupCorrection("john", "salesRep") runs before matching (resolver.ts:34). If "john → John Smith" was saved earlier, the term resolves directly and no clarify bubble appears.
Cross-field fallback (salesRep ↔ customer) can quietly resolve it. If a salesRep term has 0 rep matches but exactly 1 customer match, the resolver promotes it to the customer field instead of asking (resolver.ts:43–47). Only when that fallback also fails to land a single match does the ambiguity surface.

What to improve

The current design is deliberately scoped for a single-instance demo. These are the concrete next steps, grouped by how much they unblock — each one traces back to a limitation already noted above.

Correctness & UX

  • No way to delete a bad correction. A wrong "john → John Smith" can only be overwritten via the "No" path — there's no list/clear UI.
  • Field-scoped lookup misses obvious matches. A correction saved under customer isn't consulted for salesRep; consider a cross-field correction check.
  • The "conversation" isn't really AI — it's static matching. Apart from the initial parse, the flow is deterministic string logic: includes() matching, hardcoded ["salesRep", "customer"] field lists, and correction-detection branches spread across send() and choose() gated on a non-null base. It's hard to follow, hard to extend (a new field means editing several places), and doesn't actually reason about the dialogue. A genuine agent loop — let the LLM hold conversation state and decide when to clarify/correct — would be simpler and far more extensible.
  • No fuzzy name search — only includes(). Matching is pure substring, so a typo like "jon" or "jhon" matches nothing and the query silently fails. Fuzzy/typo-tolerant matching (e.g. trigram or edit-distance) would catch these.

Scaling & production

  • Single-instance bus + SQLite. Swap to Redis pub/sub + Postgres (see Q4) before running more than one node.
  • No auth, global corrections. Scope the Correction row by userId and add auth middleware for multi-user.
  • Static FX rates. Move hardcoded currency.ts rates to a live/daily-updated source.
  • distinctValues loads the whole column into Node. Every query runs a SELECT DISTINCT full table scan, then matchCandidates linear-scans the result in JS. With many names this is both slow (scan grows with row count) and breaks the UI (a substring can return hundreds of candidates). Push matching into the DB — WHERE name ILIKE '%t%' LIMIT 10 with a trigram index — instead of filtering in JS.

Lower-priority hardening

Test coverage for the correction edge cases. The "No" path, cross-field promotion, and correction overwrite are the trickiest flows and the most prone to regression. They're worth pinning down with tests so future changes to resolver.ts / ChatPanel.tsx stay safe.
Observability on resolution outcomes. There's no signal for how often queries hit a stored correction vs. ambiguity vs. zero-match. Logging the outcome per query would show whether correction learning is actually reducing prompts over time.
